# Codegen Split Between gen/ and internal/

## Why this split exists

`buf generate` regenerates everything under `gen/` from the proto definitions
and the template set in `proto/templates/`. The generator overwrites files on
every run: any hand-written wire format that lives in `gen/method/` or
`gen/service/` will be silently clobbered the next time someone runs codegen.

To keep hand-verified wire formats safe, hand-written code lives under
`internal/` instead. `gen/` is a pure codegen output directory and is safe to
delete and regenerate.

## Where things live

### `gen/`

Generated by `buf generate` via `protoc-gen-anything` and the templates in
`proto/templates/`. Regenerated on every `buf generate` run.

- `gen/notebooklm/v1alpha1/` — protobuf-generated Go types (`*.pb.go`).
- `gen/method/*_encoder.go` — per-method `EncodeXxxArgs` functions. The
  template emits either a generic `argbuilder.EncodeRPCArgs` call (when the
  proto has an `arg_format` annotation) or an empty TODO stub (when it does
  not). These encoders are best-effort and may produce the wrong wire format
  for commands whose real shape does not match the simple `arg_format` DSL.
- `gen/service/*_client.go` — per-service typed clients that wrap the RPC
  client and call the matching `gen/method` encoder. These are always safe to
  regenerate.

### `internal/method/`

Hand-written encoders whose wire format has been verified against a HAR
capture from the NotebookLM web client. `buf generate` never writes to this
directory. Every file in `internal/method/` MUST carry the guard comment:

    // Wire format verified against HAR capture: <path> — do not regenerate.

Where `<path>` cites the HAR capture (or dated context) that justifies the
shape, so future readers can trace the provenance without re-capturing.

A file graduates from `gen/method/` into `internal/method/` as soon as a
developer verifies the wire format against a HAR capture and replaces the
argbuilder stub with a hand-coded encoder. The old argbuilder stub stays in
`gen/method/` under the same function name — the service client (in
`gen/service/`) continues to call it, and callers that want the correct wire
format bypass the service client and invoke the `internal/method/` version
directly (typically via `rpc.Call{ID: ..., Args: intmethod.Encode...}`).

## The rule

- **No file exists in both places.** Once an encoder moves to
  `internal/method/`, delete the hand-written copy from `gen/method/`. The
  next `buf generate` will recreate a regeneratable stub with the same
  function name; that stub is deliberately distinct from the HAR-verified
  version and callers that need correctness must import
  `github.com/tmc/nlm/internal/method` (aliased as `intmethod`) explicitly.

- **Guard comment is mandatory.** Every encoder in `internal/method/` must
  carry the `// Wire format verified against HAR capture: ...` guard above
  the function (or at the top of the file, when a single capture covers
  multiple functions). Reviewers should reject any file in `internal/method/`
  without it.

- **Tests ship with the code that tests them.** Table-driven tests for
  `internal/method/` encoders live in `internal/method/*_test.go`. Golden
  payloads live in `internal/method/testdata/`.

## Verifying `buf generate` is clean

From the repo root:

    (cd proto && buf generate)
    git status -- internal/

`git status` must show zero changes under `internal/`. Any modification means
a hand-written file has been placed in a codegen output path by mistake.

## Common mistakes

- Writing a new HAR-verified encoder directly into `gen/method/`. It will be
  clobbered on the next `buf generate`. Put it in `internal/method/`.
- Forgetting the guard comment on an `internal/method/` file. Reviewers
  cannot tell at a glance whether the file is HAR-verified or just parked
  there.
- Importing `gen/method` when you want the HAR-verified encoder. You want
  `internal/method`. The two packages share symbol names on purpose: callers
  pick the one they need.
